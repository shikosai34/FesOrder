# 文化祭モバイルオーダー＆スタンプラリーシステム 詳細仕様書

## 1. システム概要

本システムは、文化祭における模擬店（サークル）の注文をスマートフォンから事前に行う「モバイルオーダー機能」と、イベント内の回遊性を高める「スタンプラリー機能」を統合したWebアプリケーションである。
注文の混雑緩和と、決済・商品受け渡しのスムーズな進行を実現し、支払い完了と同時にスタンプを付与することで参加者の満足度向上を目指す。

---

## 2. システム利用者（権限ロール定義）

システムを利用するユーザーは、以下の理論上のロールに分類され、実際のデータベースにおけるロール（`RoleType`）にマッピングされてアクセス制限が行われる。

| 理論上のロール | 実装上のロールキー (`RoleType`) | 説明 | 主な操作範囲 |
| :--- | :--- | :--- | :--- |
| **来場者** | `viewer` / ゲスト | 自分のスマホからメニューを閲覧し、注文（予約）やステータス（準備中、呼出中など）、スタンプ獲得状況を確認する。 | メニュー・注文状況の閲覧、注文作成 |
| **サークル一般店員** | `cashier`<br>`kitchen_staff`<br>`waiter`<br>`stock_manager` | 所属サークルのPOS画面や管理画面（バックヤード）を操作し、注文受付、現地決済確認、調理完了、商品受け渡し、在庫更新を行う。 | 注文ステータス更新、在庫管理 |
| **サークル代表者・会計** | `circle_manager` | 自サークルのマネージャーとして、一般店員の追加・削除（招待管理）、メニューの追加・編集・「売り切れ」操作、リアルタイム売上確認を行う。 | サークル管理、メニュー編集、スタッフ招待、売上閲覧 |
| **景品交換所スタッフ** | `viewer` または `event_admin` | 本部テント等で、来場者のスタンプラリー達成画面を確認し、景品の引き渡し（引換済フラグの更新）を行う。 | 景品引換状況の更新・確認 |
| **本部会計 / 全体管理者** | `event_admin` | 実行委員のシステム担当者。全サークルの売上状況をリアルタイムで監視・集計する。サークルの新規登録、マスターデータの管理、システム緊急停止操作を行う。 | 全機能へのアクセス、イベント管理、全サークル売上閲覧 |

## 3. 認証・ログイン仕様

### 3.1. 一般ユーザー（来場者）
* **方式**: アカウント登録（必須）による認証
* **仕組み**: 
  * デジタルリストバンドの登録（チェックイン）およびモバイルオーダー等の機能利用には、ユーザー認証（ログインまたは新規登録）が必須となっている。
  * 認証完了後に `callbackUrl` を用いて元のプロセスへシームレスにリダイレクトし、ユーザーのアカウントとリストバンドを紐付けることで、セキュアな状態管理を行う。

### 3.2. サークルスタッフ・本部運営
* **方式**: Better-Auth によるメール・パスワード（またはPIN）認証
* **仕組み**: 
  * イベント管理者やサークル代表者はメールアドレスとパスワードでログインする。
  * 現場のサークル店員として入る場合は、イベント名・サークル名を選択し、メールアドレス＋暗証番号（PIN）で簡易ログインする。この機能は `/login` ページの「店舗・スタッフ簡易」タブとして提供される。

### 3.3. 初期管理者の設定（ニワトリと卵問題の解決）
* サーバーの環境変数（`.env`）に `INITIAL_SUPER_ADMIN_EMAIL` として開発責任者等のメールアドレスを設定する。
* ログイン時のメールアドレスがこの環境変数と一致した場合、システムが自動的に `event_admin`（全体システム管理者）権限を付与してアカウントを作成する。
* 以降のサークル代表者などの権限付与は、この初期管理者が管理画面から行う。

## 4. 主要機能・運用要件

### 4.1. モバイルオーダー機能
* **メニュー閲覧・カート**: サークルごとのメニューおよびトッピングを選択し、注文を確定する。
* **ステータス遷移**:
  * `pending`（未決済/要受付）：注文確定直後、決済待ちの状態。
  * `preparing`（決済済/調理中）：屋台でPayPayまたは現金の支払いを確認後、店員がステータスを更新し、調理を開始した状態。
  * `ready`（調理完了/呼出中）：品物が完成し、ユーザーに「受け取りに来てください」と通知・表示される状態。
  * `completed`（受渡済）：商品の引き渡しが完了した状態（`completed: true`）。
  * `cancelled`（キャンセル）：無断キャンセルや機材トラブルによる手動キャンセル状態。

### 4.2. スタンプラリー機能
* **スタンプカード表示**: ユーザー画面に現在のスタンプ獲得状況を表示。
* **自動スタンプ付与**: オーダーのステータスが `pending`（未決済） ➔ `preparing`（決済済）になったタイミングで、システムが自動的にそのサークルのスタンプ獲得レコードを作成する。
* **景品交換**: 規定数のスタンプを集めたユーザーに対し、景品交換所のスタッフが確認後、「景品引換済」フラグを立てる（二重受け取り防止）。

### 4.3. 運用・例外ルール
* **無断キャンセル対応**: 一定時間（例: 30分）経過しても決済に現れない `pending`（未決済）の注文は、サークル代表者または本部の判断で管理画面から手動キャンセルを行う。
* **返金フロー**: `preparing`（決済済）の後に機材トラブル等で提供できなくなった場合、システム上のステータスを `cancelled` に変更した上で、現地でのPayPayアプリからの返金操作または現金での返金を行う運用とする。
* **売り切れ管理**: 在庫数の厳密な自動カウントダウンは行わず、サークル代表者が管理画面から手動で `soldOut`（`is_sold_out`）フラグを切り替える運用とする。

## 5. データモデル（主要テーブル設計）

本システムは SQLite を採用し、Drizzle ORM を用いてスキーマ定義を行っている。

### 5.1. ユーザー・認証関連 (Better-Auth 準拠)

#### `user`（ユーザー情報）
* `id` (PK, text): ユーザー識別用一意識別子
* `name` (text): 表示名
* `email` (text, Unique): メールアドレス
* `emailVerified` (integer/boolean): メール認証済フラグ
* `image` (text, Nullable): アイコン画像パス
* `createdAt` (integer/timestamp): 作成日時
* `updatedAt` (integer/timestamp): 更新日時

#### `session`（セッション情報）
* `id` (PK, text)
* `userId` (FK -> `user.id`): ユーザー参照
* `token` (text, Unique)
* `expiresAt` (integer/timestamp)
* `ipAddress` (text, Nullable)
* `userAgent` (text, Nullable)
* `createdAt`, `updatedAt`

---

### 5.2. サークル・イベント・スタッフ関連

#### `event`（イベント情報）
* `id` (PK, text)
* `eventName` (text): イベント名（例: 「第34回 志高祭」）
* `description` (text, Nullable): 詳細説明
* `startDate` (integer/timestamp, Nullable)
* `endDate` (integer/timestamp, Nullable)
* `createdAt`, `updatedAt`

#### `circle`（サークル情報）
* `id` (PK, text)
* `eventId` (FK -> `event.id`): 所属イベント
* `name` (text): サークル名 / 模擬店名
* `description` (text, Nullable): サークル紹介
* `password` (text): 旧サークル専用パスワード（後方互換性維持のためランダムに自動生成された文字列をハッシュ化して格納）
* `iconImagePath` (text, Nullable): アイコン画像パス
* `backgroundImagePath` (text, Nullable): 背景画像パス
* `createdAt`, `updatedAt`

#### `membership`（スタッフ/管理者メンバーシップ）
* `id` (PK, text)
* `userEmail` (text): スタッフのメールアドレス
* `userName` (text): スタッフの氏名
* `circleId` (FK -> `circle.id`, Nullable): 所属サークル（イベント管理者の場合はNULL）
* `eventId` (FK -> `event.id`, Nullable): 所属イベント
* `role` (text): 付与ロール（`circle_manager`, `cashier`, `kitchen_staff` 等）
* `pin` (text, Nullable): 簡易認証用PINコード
* `isActive` (integer/boolean): アクティブ状態フラグ
* `invitedAt` (integer/timestamp, Nullable)
* `acceptedAt` (integer/timestamp, Nullable)
* `createdAt`, `updatedAt`

#### `invite_token`（スタッフ招待トークン）
* `id` (PK, text)
* `token` (text, Unique): 招待用トークン文字列
* `circleId` (FK -> `circle.id`, Nullable)
* `eventId` (FK -> `event.id`, Nullable)
* `role` (text): 招待時に付与するロール
* `maxUses` (integer): 最大使用可能人数
* `usedCount` (integer): 現在の使用人数
* `expiresAt` (integer/timestamp): 有効期限
* `createdBy` (text): 作成者のメールアドレス
* `createdAt` (integer/timestamp)

#### `menu`（メニュー情報）
* `id` (PK, text)
* `circleId` (FK -> `circle.id`): 提供元サークル
* `name` (text): 商品名
* `price` (integer): 金額
* `imagePath` (text): 商品画像パス
* `description` (text, Nullable): 商品説明
* `additionalInfo` (text, Nullable): 補足情報（アレルギー等）
* `soldOut` (integer/boolean): 売り切れフラグ（デフォルト: false）
* `stockQuantity` (integer): 在庫数（簡易管理用、デフォルト: 0）
* `createdAt`, `updatedAt`

#### `topping`（トッピング情報）
* `id` (PK, text)
* `circleId` (FK -> `circle.id`): 提供元サークル
* `name` (text): トッピング名
* `price` (integer): トッピング金額
* `description` (text, Nullable)
* `soldOut` (integer/boolean): 売り切れフラグ
* `createdAt`, `updatedAt`

#### `menu_topping`（メニューとトッピングの紐付け）
* `id` (PK, text)
* `menuId` (FK -> `menu.id`)
* `toppingId` (FK -> `topping.id`)
* `createdAt`

### 5.4. オーダー関連

#### `orders`（注文ヘッダー）
* `id` (PK, text)
* `circleId` (FK -> `circle.id`): 注文先サークル
* `orderNumber` (text, Unique): 来場者に提示するわかりやすい注文番号
* `peopleCount` (integer): 来客人数（または数量）
* `totalPrice` (integer): 注文の合計金額
* `status` (text): ステータス（`pending`, `preparing`, `ready`, `completed`, `cancelled`）
* `completed` (integer/boolean): 完了フラグ
* `completedAt` (integer/timestamp, Nullable): 受け渡し完了日時
* `estimatedTime` (integer, Nullable): 提供予測時間（分）
* `cashierId` (text, Nullable): 担当レジスタッフID
* `createdAt`, `updatedAt`

#### `order_item`（注文明細）
* `id` (PK, text)
* `orderId` (FK -> `orders.id`)
* `menuId` (FK -> `menu.id`)
* `menuName` (text): 注文時のメニュー名（メニュー名変更時の履歴保存用スナップショット）
* `menuPrice` (integer): 注文時の単価（価格変更時の履歴保存用スナップショット）
* `quantity` (integer): 数量
* `createdAt`

#### `order_item_topping`（注文明細のトッピング）
* `id` (PK, text)
* `orderItemId` (FK -> `order_item.id`)
* `toppingId` (FK -> `topping.id`)
* `toppingName` (text): 注文時のトッピング名（スナップショット）
* `toppingPrice` (integer): 注文時のトッピング価格（スナップショット）
* `createdAt`

## 6. 重要なビジネスロジック：権限の委譲フロー

模擬店の管理コストを最小限に抑えるため、スタッフの追加は以下の「自律分散型」フローで行う。

1. **イベント管理者（本部）の作業**:
   * `event_admin` 権限を持つユーザーは、事前に各サークルの「代表者1名のメールアドレス」をシステムに登録し、そのサークルに対して `circle_manager`（サークル代表者・会計）ロールの `membership` を作成する。
2. **サークル代表者のログインと招待トークン生成**:
   * サークル代表者がシステムにログインすると、ダッシュボードに「自サークルのスタッフ管理」メニューが表示される。
   * 代表者はそこから「スタッフ招待用リンク」を生成する（バックエンドで `invite_token` レコードが作成される）。この際、付与するロール（`cashier`, `kitchen_staff` 等）を指定する。
3. **スタッフ（店員）の登録**:
   * 追加したいメンバーに対し招待リンクを共有し、メンバーが学校のGoogleアカウントでアクセスすると、自動的にそのサークルの `membership`（指定されたロール）が作成され、すぐにPOS画面やバックヤード画面が利用可能になる。

## 7. 現状の実装状況とギャップ分析（実装済み vs 未実装）

現在リポジトリに実装されているソースコードと、詳細仕様書の内容を比較した実装状況です。

### 7.1. 実装済み機能 (Implemented)
* **イベント管理・サークル管理 (管理者用UI & API)**:
  * フロントエンド: `apps/web/src/app/admin/page.tsx` でのイベント・サークルの作成・削除 UI。
  * バックエンド: `apps/server/src/routes/event.ts` および `circle.ts` でのイベント・サークルのCRUD機能。
* **モバイルオーダー機能**:
  * メニュー閲覧、トッピング選択、カート追加、注文作成機能（`apps/web/src/app/menu/page.tsx`、`apps/server/src/routes/menu.ts`, `order.ts`）。
* **店舗・スタッフ簡易ログイン (PIN認証)**:
  * イベント名・サークル名を選択し、メールアドレスとPINでログインするスタッフ用簡易ログイン機能（`/login` の「店舗・スタッフ簡易」タブに統合、`/circle-login` は `/login` にリダイレクト）。
* **バックヤード・POS機能 (厨房管理・レジ管理)**:
  * 注文一覧のリアルタイム監視、調理ステータス（準備中・調理中・呼出中・受渡済）の更新機能。
* **権限ロール定義とバリデーションの一貫性**:
  * DBスキーマ、フロントエンド、バックエンドのバリデーションですべて `event_admin`, `circle_manager`, `cashier`, `kitchen_staff`, `waiter`, `stock_manager`, `viewer` に統一されました。
* **メンバー追加・招待受入API**:
  * フロントエンドとバックエンドの双方で、メンバーの追加や招待の受入時に `userEmail` と `userName` を正しく送受信し、メンバーシップを登録できるように修正されました。
* **管理者機能における権限検証**:
  * バックエンドのイベント作成・削除（`POST/DELETE /api/events`）およびサークル作成・削除（`POST/DELETE /api/circles`）に、`getAdminSession` を用いた管理者（`event_admin`）権限チェックが適用されています。
* **初期管理者（`INITIAL_SUPER_ADMIN_EMAIL`）の自動昇格ロジック**:
  * ログイン（セッション取得）時に環境変数の `INITIAL_SUPER_ADMIN_EMAIL` に設定したメールアドレスと同じアドレスの場合、自動的に `event_admin` ロールを付与する処理が `apps/server/src/utils/auth.ts` に実装されました。
* **認証の必須化とリストバンド連携**:
  * ゲストアクセスを廃止し、来場者のチェックインおよびオーダーにアカウント登録（ログイン）を必須化。
* **権限階層と表示の最適化**:
  * `event_admin` と `circle_manager` を兼務するユーザーの認証フローを修正し、管理者環境とサークル環境へのシームレスなアクセスを実現。また、UI上の権限を「在庫編集」など分かりやすい日本語ラベルで表示。
* **UI/UXの刷新 (RawBlockデザイン)**:
  * アプリケーション全体に「RawBlock（ブルータリズム）」デザインシステムを導入。角丸を排除し、高コントラストでモノスペースフォントを基調としたレスポンシブUIへ移行済み。QRスキャナーなどのPOS操作画面もモバイル・タブレットに最適化。

### 7.2. 設計・実装に乖離（ズレ）がある機能（修正が必要）
* 現在、以前指摘されていた主な乖離はすべて解消され、設計に沿った実装となっています。

### 7.3. 未実装機能 (Missing)
* **スタンプラリー機能全般**:
  * DBスキーマにスタンプ・景品交換に関するテーブル（`Stamps`, `UserStamps`, `RewardRedemptions` 等）が定義されていません。
  * バックエンドにスタンプラリー関連のAPIルートがありません。
  * フロントエンドにスタンプ獲得状況を表示するUIや、景品交換用スタッフ画面がありません。
